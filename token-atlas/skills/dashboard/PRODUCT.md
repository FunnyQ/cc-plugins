# Product

## Register

product

## Users

Claude Code power users who want a local, private view of usage, cost, model mix, and project activity. They are usually checking spend, reviewing work rhythm, or debugging why usage changed across recent sessions.

## Product Purpose

The dashboard turns local Claude Code data into a scannable operational view. Success means Q can open it, understand the current usage window in seconds, compare models and projects without reading raw JSON, and trust that the numbers are local and practical.

## Brand Personality

Precise, composed, quietly technical. The interface should feel like a serious instrument for a developer, not a marketing analytics page. Nordic mythology may inform the atmosphere, but only through structure, rhythm, naming, and material restraint.

## Anti-references

Avoid flat admin boilerplate, generic SaaS card grids, neon-on-black observability cliches, glassmorphism, decorative gradients, novelty controls that slow down reading the data, and literal fantasy UI. No Viking cosplay, rune stickers, medieval props, ornate knotwork, or mythology references that compete with the operational task.

## Nordic Mythology Inflection

Use Nordic mythology as structural atmosphere, not decoration. The dashboard may borrow from ideas of navigation, runes, cold metal, carved records, branching worlds, and ritual accounting, but it should never feel like a game UI or fantasy interface.

The visual language should suggest:

- wayfinding over ornament: axes, routes, ledgers, thresholds
- cold clarity: arctic neutrals, low-chroma blue-gray, iron, slate, frost
- restrained mythic motifs: rune-like tick marks, carved dividers, world-tree hierarchy, saga-style chronology
- cost as offering: usage and spend should feel measured, traceable, and consequential
- local trust as kept record: private data feels like a sealed ledger, not cloud analytics

## Design Principles

1. Put the working question first: what changed, where tokens went, and what it cost.
2. Use density with rhythm, giving charts and tables enough structure without burying the metrics.
3. Make local trust visible through restrained copy and clear metadata, not privacy theater.
4. Let color encode model, state, and intensity; do not use it as decoration.
5. Keep the interface calm enough for repeated use.
6. Let Nordic references live in structure, rhythm, and naming, not literal decoration.

## Accessibility & Inclusion

Target WCAG AA contrast, keyboard-visible focus states, reduced-motion compatibility, and data encodings that do not rely on hue alone.
